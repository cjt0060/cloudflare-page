## Introduction

Building highly available systems requires thinking beyond single-region deployments. In this article, I'll walk through the architectural patterns and implementation details for building a multi-region AWS architecture that achieves 99.99% uptime.

## Why Multi-Region?

Single-region architectures are vulnerable to regional outages. While AWS regions are designed for high availability with multiple Availability Zones, entire regions can experience degraded performance. For mission-critical workloads, multi-region is the gold standard.

### Key Benefits

- **Reduced latency** for globally distributed users
- **Disaster recovery** with near-zero RPO/RTO
- **Compliance** with data residency requirements
- **Blast radius reduction** for infrastructure failures

## Architecture Overview

The architecture uses an active-active pattern across two AWS regions:

```
┌─────────────────────────────────────────────┐
│              Route 53 (Global)               │
│         Latency-based routing +              │
│            health checks                     │
└──────────────┬──────────────┬───────────────┘
               │              │
    ┌──────────▼──────┐  ┌───▼──────────────┐
    │   us-east-1     │  │    eu-west-1     │
    │                 │  │                  │
    │  ALB + EKS      │  │  ALB + EKS      │
    │  Aurora Global  │  │  Aurora Replica  │
    │  ElastiCache    │  │  ElastiCache     │
    └─────────────────┘  └──────────────────┘
```

## Route 53 Configuration

Route 53 serves as the global traffic manager with latency-based routing and health checks:

```hcl
resource "aws_route53_health_check" "primary" {
  fqdn              = "api-us-east-1.example.com"
  port              = 443
  type              = "HTTPS"
  resource_path     = "/health"
  failure_threshold = "3"
  request_interval  = "10"

  tags = {
    Name = "primary-health-check"
  }
}

resource "aws_route53_record" "api" {
  zone_id = aws_route53_zone.main.zone_id
  name    = "api.example.com"
  type    = "A"

  latency_routing_policy {
    region = "us-east-1"
  }

  set_identifier = "us-east-1"

  alias {
    name                   = aws_lb.primary.dns_name
    zone_id                = aws_lb.primary.zone_id
    evaluate_target_health = true
  }

  health_check_id = aws_route53_health_check.primary.id
}
```

## Data Replication Strategy

Aurora Global Database provides cross-region replication with less than 1 second of lag:

```hcl
resource "aws_rds_global_cluster" "main" {
  global_cluster_identifier = "global-cluster"
  engine                    = "aurora-postgresql"
  engine_version            = "15.4"
  database_name             = "application"
}

resource "aws_rds_cluster" "primary" {
  cluster_identifier        = "primary-cluster"
  engine                    = "aurora-postgresql"
  global_cluster_identifier = aws_rds_global_cluster.main.id
  master_username           = "admin"
  manage_master_user_password = true
}

resource "aws_rds_cluster" "secondary" {
  provider                  = aws.secondary
  cluster_identifier        = "secondary-cluster"
  engine                    = "aurora-postgresql"
  global_cluster_identifier = aws_rds_global_cluster.main.id

  depends_on = [aws_rds_cluster.primary]
}
```

## Failover Automation

Automated failover reduces recovery time from minutes to seconds:

```python
import boto3

def lambda_handler(event, context):
    """Automated failover triggered by CloudWatch alarm."""
    rds = boto3.client('rds', region_name='eu-west-1')

    response = rds.failover_global_cluster(
        GlobalClusterIdentifier='global-cluster',
        TargetDbClusterIdentifier='arn:aws:rds:eu-west-1:123456789:cluster:secondary-cluster'
    )

    # Update Route 53 to route all traffic to secondary
    route53 = boto3.client('route53')
    route53.change_resource_record_sets(
        HostedZoneId='Z1234567890',
        ChangeBatch={
            'Changes': [{
                'Action': 'UPSERT',
                'ResourceRecordSet': {
                    'Name': 'api.example.com',
                    'Type': 'A',
                    'AliasTarget': {
                        'HostedZoneId': 'Z0987654321',
                        'DNSName': 'alb-secondary.eu-west-1.elb.amazonaws.com',
                        'EvaluateTargetHealth': True
                    }
                }
            }]
        }
    )

    return {'statusCode': 200, 'body': 'Failover complete'}
```

## Monitoring and Observability

Cross-region monitoring requires a centralized observability stack:

- **CloudWatch Cross-Region Dashboard** for unified metrics
- **Route 53 Health Check alarms** for immediate failover triggers
- **Aurora Global Database lag monitoring** to detect replication issues
- **Custom latency metrics** comparing response times across regions

## Cost Considerations

Multi-region doubles many costs but provides strategies for optimization:

1. **Right-size secondary region** — run at lower capacity during normal operations
2. **Reserved Instances** — commit to baseline capacity in both regions
3. **Data transfer optimization** — compress replication streams
4. **Spot instances** — use for non-critical workloads in secondary

## Lessons Learned

After operating this architecture in production for two years:

1. **Test failovers regularly** — monthly game days prevent surprise failures
2. **Session management** — use distributed session stores (DynamoDB Global Tables)
3. **DNS TTL matters** — keep TTLs low (60s) for faster failover
4. **Eventual consistency** — design application logic to handle replication lag

## Conclusion

Multi-region architectures on AWS provide the highest levels of availability for mission-critical applications. While they add complexity and cost, the resilience benefits are essential for workloads that require 99.99%+ uptime.
