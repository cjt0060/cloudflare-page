## Introduction

Managing Terraform at scale is fundamentally different from managing a few dozen resources. When your organization has hundreds of deployments across multiple environments and teams, you need robust module architecture, automated testing, and state management strategies.

## Module Architecture

### The Module Hierarchy

Structure modules in a three-tier hierarchy:

```
terraform/
├── modules/               # Reusable building blocks
│   ├── networking/
│   │   ├── vpc/
│   │   ├── subnet/
│   │   └── security-group/
│   ├── compute/
│   │   ├── eks-cluster/
│   │   ├── ec2-instance/
│   │   └── lambda-function/
│   └── data/
│       ├── rds-cluster/
│       ├── dynamodb-table/
│       └── s3-bucket/
├── compositions/          # Combine modules into solutions
│   ├── web-application/
│   ├── data-pipeline/
│   └── kubernetes-platform/
└── deployments/           # Environment-specific configs
    ├── production/
    ├── staging/
    └── development/
```

### Module Interface Design

Design module interfaces like APIs — stable, versioned, and well-documented:

```hcl
# modules/compute/eks-cluster/variables.tf
variable "cluster_name" {
  type        = string
  description = "Name of the EKS cluster"

  validation {
    condition     = can(regex("^[a-z][a-z0-9-]{2,39}$", var.cluster_name))
    error_message = "Cluster name must be lowercase alphanumeric with hyphens, 3-40 chars."
  }
}

variable "kubernetes_version" {
  type        = string
  description = "Kubernetes version for the EKS cluster"
  default     = "1.28"

  validation {
    condition     = contains(["1.27", "1.28", "1.29"], var.kubernetes_version)
    error_message = "Supported versions: 1.27, 1.28, 1.29."
  }
}

variable "node_groups" {
  type = map(object({
    instance_types = list(string)
    min_size       = number
    max_size       = number
    desired_size   = number
    labels         = optional(map(string), {})
    taints = optional(list(object({
      key    = string
      value  = string
      effect = string
    })), [])
  }))
  description = "Map of node group configurations"
}

variable "enable_cluster_autoscaler" {
  type        = bool
  default     = true
  description = "Whether to install Cluster Autoscaler"
}
```

### Composition Pattern

Compositions wire together modules with sensible defaults:

```hcl
# compositions/kubernetes-platform/main.tf
module "vpc" {
  source = "../../modules/networking/vpc"

  name               = "${var.environment}-${var.cluster_name}"
  cidr               = var.vpc_cidr
  availability_zones = var.availability_zones
  
  enable_nat_gateway = true
  single_nat_gateway = var.environment != "production"
}

module "eks" {
  source = "../../modules/compute/eks-cluster"

  cluster_name       = var.cluster_name
  kubernetes_version = var.kubernetes_version
  vpc_id             = module.vpc.vpc_id
  subnet_ids         = module.vpc.private_subnet_ids
  node_groups        = var.node_groups

  enable_cluster_autoscaler = true
}

module "monitoring" {
  source = "../../modules/observability/prometheus-stack"

  cluster_name    = module.eks.cluster_name
  cluster_endpoint = module.eks.cluster_endpoint
  
  enable_grafana     = true
  retention_days     = var.environment == "production" ? 90 : 30
  storage_class_name = "gp3"
}
```

## Testing with Terratest

### Unit Tests

Test individual module behavior:

```go
package test

import (
    "testing"

    "github.com/gruntwork-io/terratest/modules/terraform"
    "github.com/stretchr/testify/assert"
)

func TestVPCModule(t *testing.T) {
    t.Parallel()

    terraformOptions := &terraform.Options{
        TerraformDir: "../modules/networking/vpc",
        Vars: map[string]interface{}{
            "name":               "test-vpc",
            "cidr":               "10.0.0.0/16",
            "availability_zones": []string{"us-east-1a", "us-east-1b"},
            "enable_nat_gateway":  true,
            "single_nat_gateway":  true,
        },
    }

    defer terraform.Destroy(t, terraformOptions)
    terraform.InitAndApply(t, terraformOptions)

    vpcID := terraform.Output(t, terraformOptions, "vpc_id")
    assert.NotEmpty(t, vpcID)

    privateSubnets := terraform.OutputList(t, terraformOptions, "private_subnet_ids")
    assert.Equal(t, 2, len(privateSubnets))
}
```

### Integration Tests

Test compositions end-to-end:

```go
func TestKubernetesPlatform(t *testing.T) {
    t.Parallel()

    terraformOptions := &terraform.Options{
        TerraformDir: "../compositions/kubernetes-platform",
        Vars: map[string]interface{}{
            "environment":    "test",
            "cluster_name":   fmt.Sprintf("test-%s", random.UniqueId()),
            "vpc_cidr":       "10.99.0.0/16",
            "node_groups": map[string]interface{}{
                "default": map[string]interface{}{
                    "instance_types": []string{"t3.medium"},
                    "min_size":       1,
                    "max_size":       3,
                    "desired_size":   2,
                },
            },
        },
    }

    defer terraform.Destroy(t, terraformOptions)
    terraform.InitAndApply(t, terraformOptions)

    kubeconfig := terraform.Output(t, terraformOptions, "kubeconfig")
    assert.NotEmpty(t, kubeconfig)

    // Verify cluster is accessible
    clusterEndpoint := terraform.Output(t, terraformOptions, "cluster_endpoint")
    assert.Contains(t, clusterEndpoint, "eks.amazonaws.com")
}
```

## State Management

### Remote State with S3

```hcl
terraform {
  backend "s3" {
    bucket         = "company-terraform-state"
    key            = "production/kubernetes-platform/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"
    encrypt        = true
  }
}
```

### State Isolation Strategy

- **Per-environment backends** prevent cross-environment accidents
- **Per-component state files** reduce blast radius and improve plan times
- **State locking** with DynamoDB prevents concurrent modifications
- **State encryption** protects sensitive outputs

## CI/CD Pipeline

Automate Terraform workflows with GitHub Actions:

```yaml
name: Terraform
on:
  pull_request:
    paths: ["terraform/**"]

jobs:
  plan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3

      - name: Terraform Init
        run: terraform init
        working-directory: terraform/deployments/staging

      - name: Terraform Validate
        run: terraform validate

      - name: Terraform Plan
        run: terraform plan -out=plan.tfplan
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}

      - name: Post Plan to PR
        uses: actions/github-script@v7
        with:
          script: |
            const output = `#### Terraform Plan 📖
            \`\`\`
            ${process.env.PLAN_OUTPUT}
            \`\`\``;
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: output
            });
```

## Drift Detection

Regularly detect and alert on infrastructure drift:

```bash
#!/bin/bash
# Run as a scheduled job
terraform plan -detailed-exitcode -out=drift.plan 2>&1

EXIT_CODE=$?
if [ $EXIT_CODE -eq 2 ]; then
  echo "DRIFT DETECTED"
  # Send alert to Slack/PagerDuty
fi
```

## Key Takeaways

1. **Module versioning** — use semantic versioning and a private registry
2. **Test everything** — unit tests for modules, integration tests for compositions
3. **Small state files** — decompose into logical components
4. **Policy as code** — use Sentinel or OPA to enforce standards
5. **Automate workflows** — plan on PR, apply on merge to main
6. **Detect drift** — scheduled plans catch manual changes

## Conclusion

Terraform at scale requires treating your infrastructure code with the same rigor as application code. Invest in module architecture, testing, and automation early — the compounding returns are substantial.
