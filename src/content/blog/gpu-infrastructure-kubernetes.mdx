## Introduction

The explosion of AI/ML workloads has created unprecedented demand for GPU infrastructure. Kubernetes has become the standard orchestration platform for these workloads, but managing GPU resources requires specialized knowledge beyond standard cluster operations.

## GPU Node Configuration

### NVIDIA GPU Operator

The GPU Operator automates driver installation and device plugin management:

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: gpu-operator
---
# Install via Helm
# helm install gpu-operator nvidia/gpu-operator \
#   --namespace gpu-operator \
#   --set driver.enabled=true \
#   --set toolkit.enabled=true \
#   --set devicePlugin.enabled=true \
#   --set mig.strategy=mixed
```

### Node Labels and Taints

```yaml
apiVersion: v1
kind: Node
metadata:
  name: gpu-node-01
  labels:
    nvidia.com/gpu.product: NVIDIA-A100-SXM4-80GB
    nvidia.com/gpu.count: "8"
    node.kubernetes.io/instance-type: p4d.24xlarge
    topology.kubernetes.io/zone: us-east-1a
spec:
  taints:
    - key: nvidia.com/gpu
      value: "true"
      effect: NoSchedule
```

## GPU Scheduling Strategies

### Basic GPU Request

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: training-job
spec:
  containers:
    - name: trainer
      image: nvcr.io/nvidia/pytorch:24.01-py3
      resources:
        limits:
          nvidia.com/gpu: 4
      volumeMounts:
        - name: training-data
          mountPath: /data
        - name: dshm
          mountPath: /dev/shm
  volumes:
    - name: training-data
      persistentVolumeClaim:
        claimName: training-dataset
    - name: dshm
      emptyDir:
        medium: Memory
        sizeLimit: 64Gi
  tolerations:
    - key: nvidia.com/gpu
      operator: Exists
      effect: NoSchedule
  nodeSelector:
    nvidia.com/gpu.product: NVIDIA-A100-SXM4-80GB
```

### Multi-Instance GPU (MIG)

Partition A100 GPUs for efficient utilization of inference workloads:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: inference-server
spec:
  containers:
    - name: model-server
      image: nvcr.io/nvidia/tritonserver:24.01-py3
      resources:
        limits:
          nvidia.com/mig-3g.40gb: 1
      env:
        - name: MODEL_REPOSITORY
          value: /models
```

### Topology-Aware Scheduling

For multi-GPU training, NVLink topology matters:

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: gpu-training-high
value: 1000000
globalDefault: false
description: "High priority for GPU training jobs"
---
apiVersion: batch/v1
kind: Job
metadata:
  name: distributed-training
spec:
  parallelism: 4
  completions: 4
  template:
    metadata:
      annotations:
        scheduling.x-k8s.io/topology-request: "nvidia.com/gpu-same-node"
    spec:
      schedulerName: topology-aware-scheduler
      containers:
        - name: worker
          image: training-image:latest
          resources:
            limits:
              nvidia.com/gpu: 8
          env:
            - name: NCCL_SOCKET_IFNAME
              value: "eth0"
            - name: NCCL_DEBUG
              value: "INFO"
```


## Monitoring GPU Resources

### DCGM Exporter for Prometheus

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: dcgm-exporter
  namespace: monitoring
spec:
  selector:
    matchLabels:
      app: dcgm-exporter
  endpoints:
    - port: metrics
      interval: 15s
---
# Key metrics to monitor:
# DCGM_FI_DEV_GPU_UTIL - GPU utilization %
# DCGM_FI_DEV_MEM_COPY_UTIL - Memory utilization %
# DCGM_FI_DEV_GPU_TEMP - GPU temperature
# DCGM_FI_DEV_POWER_USAGE - Power consumption
# DCGM_FI_DEV_PCIE_REPLAY_COUNTER - PCIe errors
```

### Grafana Dashboard Query


## Cost Optimization

### Spot/Preemptible Instances for Training

```hcl
resource "google_container_node_pool" "gpu_spot" {
  name       = "gpu-spot-pool"
  cluster    = google_container_cluster.ml.name
  
  autoscaling {
    min_node_count = 0
    max_node_count = 32
  }

  node_config {
    machine_type = "a2-highgpu-8g"

    guest_accelerator {
      type  = "nvidia-tesla-a100"

    }

    taint {
      key    = "cloud.google.com/gke-spot"
      value  = "true"

  }
```

## Best Practices

1. **Right-size GPU instances** — profile workloads before provisioning
2. **Use MIG for inference** — partition expensive GPUs for smaller models
3. **Spot instances for training** — implement checkpointing for preemption
4. **Monitor GPU utilization** — idle GPUs waste significant budget
5. **Separate training and inference** — different scaling and availability needs
6. **Implement queueing** — use Kueue or Volcano for fair scheduling

## Conclusion

GPU infrastructure on Kubernetes requires specialized tooling for scheduling, monitoring, and cost management. Invest in proper GPU operators, topology-aware scheduling, and observability from day one. The cost of idle GPUs makes optimization critical at scale.
